# Android 真机测试交接文档

这份文档用于把当前 Android MVP 交给别人做真机测试。

适用对象有两类：

- 开发者：负责启动后端、打包 APK、把安装包发给测试者
- 测试者：负责在真实安卓手机上安装 APK 并按步骤验证流程

## 1. 当前要测试什么

当前 Android 客户端已经覆盖这些 MVP 流程：

- 开发登录
- Onboarding
- 目标创建
- 今日任务页
- 搭子聊天页
- 任务详情页
- 标记完成
- 任务改期
- 图片打卡上传

## 2. 先确定测试方式

真机测试分两种情况：

### 方式 A：同一局域网测试

适用条件：

- 后端跑在开发者电脑上
- 测试手机和开发者电脑连的是同一个 Wi-Fi

这是最简单的方案，优先推荐。

### 方式 B：远程测试

适用条件：

- 测试者不在你身边
- 测试手机无法直接访问你电脑所在局域网

注意：
如果是远程测试，当前本地后端默认不能直接给别人用，必须先让后端具备公网可访问能力。

可以用这些方式之一：

- 把后端部署到服务器
- 用 `cloudflared`、`ngrok` 之类的隧道工具暴露本地服务

如果没做这一步，远程测试者即使装上 APK，也无法正常登录和请求接口。

## 3. 开发者在发包前要做的事

### 第 1 步：启动后端

在 `backend/` 目录下运行：

```powershell
$env:PYTHONPATH='.'
& 'C:\Users\曾景麟\AppData\Local\Programs\Python\Python312\python.exe' -m uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

重点：

- 必须用 `--host 0.0.0.0`
- 不要只用 `127.0.0.1`

因为真机要通过网络访问你的电脑，后端必须对局域网开放。

### 第 2 步：先在电脑上验证后端正常

在电脑浏览器打开：

```text
http://127.0.0.1:8000/health
```

期望结果：

```json
{"status":"ok"}
```

### 第 3 步：确定手机应该访问哪个后端地址

#### 如果是方式 A：同一局域网

在 Windows 终端执行：

```powershell
ipconfig
```

找到当前 Wi-Fi 对应的 IPv4 地址，例如：

```text
192.168.1.23
```

那么手机要访问的后端地址就是：

```text
http://192.168.1.23:8000/api/v1/
```

#### 如果是方式 B：远程测试

必须使用公网可访问地址，例如：

```text
https://your-public-domain.example.com/api/v1/
```

或者：

```text
https://xxxxx.ngrok-free.app/api/v1/
```

### 第 4 步：修改 Android 端接口地址

打开：

[android/app/build.gradle.kts](</d:/HuaweiMoveData/Users/曾景麟/Desktop/app_agent/ai_buddy_specs/android/app/build.gradle.kts>)

找到这一行：

```kotlin
buildConfigField("String", "API_BASE_URL", "\"http://10.0.2.2:8000/api/v1/\"")
```

把它改成真实手机能访问的地址。

同一局域网示例：

```kotlin
buildConfigField("String", "API_BASE_URL", "\"http://192.168.1.23:8000/api/v1/\"")
```

远程测试示例：

```kotlin
buildConfigField("String", "API_BASE_URL", "\"https://your-public-domain.example.com/api/v1/\"")
```

### 第 5 步：重新打包 APK

在 `android/` 目录下运行：

```powershell
$env:JAVA_HOME="C:\Program Files\Eclipse Adoptium\jdk-17.0.18.8-hotspot"
$env:ANDROID_HOME="D:\SDK"
$env:ANDROID_SDK_ROOT="D:\SDK"
.\gradlew.bat assembleDebug --no-daemon
```

APK 输出路径：

[app-debug.apk](</d:/HuaweiMoveData/Users/曾景麟/Desktop/app_agent/ai_buddy_specs/android/app/build/outputs/apk/debug/app-debug.apk>)

### 第 6 步：把 APK 发给测试者

可以通过这些方式发送：

- 微信文件传输
- QQ
- 网盘
- USB 拷贝

同时建议一起发给测试者：

- 这份测试文档
- 手机浏览器要打开的 `/health` 地址
- 当前测试日期
- 当前代码提交号（如果你希望更方便定位问题）

## 4. 测试者在手机上要做的事

这部分不需要 Android Studio。

### 第 1 步：连上正确的网络

如果是方式 A：

- 手机必须和开发者电脑连同一个 Wi-Fi

如果是方式 B：

- 正常联网即可，但前提是开发者已经准备好公网后端地址

### 第 2 步：先用手机浏览器测后端

在手机浏览器打开：

```text
http://192.168.1.23:8000/health
```

或者开发者给你的公网地址对应的 `/health`。

期望结果：

```json
{"status":"ok"}
```

如果这里打不开，就先不要继续装 App，先把问题反馈给开发者。因为这说明网络本身还没通。

### 第 3 步：允许安装 APK

如果手机提示不允许安装：

1. 打开“设置”
2. 搜索“安装未知应用”或“允许未知来源安装”
3. 给文件管理器或当前打开 APK 的应用授权

### 第 4 步：安装 APK

1. 点击 APK 文件
2. 安装
3. 打开 App

## 5. 推荐测试流程

建议测试者按下面顺序走，这样最容易定位问题。

### 测试 1：登录

操作：

- 打开 App
- 输入昵称
- 点击登录

期望：

- 成功进入 Onboarding
- 不闪退
- 没有网络报错

### 测试 2：Onboarding

操作：

- 选择搭子风格
- 选择严格程度
- 填目标名称
- 填目标描述
- 填截止日期
- 填一个专注时段
- 可选填一个固定安排
- 点击生成计划

期望：

- 提交成功
- 跳转到首页
- 页面不空白

### 测试 3：首页

操作：

- 查看首页

期望：

- 能看到今日任务
- 能看到统计区域
- 布局没有明显重叠或错位

### 测试 4：进入任务详情

操作：

- 点击一条任务

期望：

- 能进入任务详情页
- 能看到时间
- 能看到打卡提示

### 测试 5：标记完成

操作：

- 输入一段完成说明
- 点击完成按钮

期望：

- 任务状态更新
- 不闪退

### 测试 6：任务改期

操作：

- 在改期输入框写原因
- 点击改期按钮

建议输入：

```text
今晚临时有事，原来的时间做不了。
```

期望：

- 页面出现搭子回复
- 出现新的安排时间
- 不会卡死

### 测试 7：聊天

操作：

- 打开聊天页
- 发一句消息

建议输入：

```text
我今晚真的不太想做这个了。
```

期望：

- 能收到回复
- 如有快捷按钮，能够显示

### 测试 8：图片打卡

操作：

- 打开某个任务详情页
- 选择一张图片
- 提交打卡

期望：

- 上传成功
- 出现审核结果
- 打开图片选择器时不闪退

## 8. 最常见的问题和排查方法

### 问题：App 一打开就提示网络错误

常见原因：

- 后端没启动
- 手机访问不到后端地址
- `API_BASE_URL` 还是模拟器地址 `10.0.2.2`
- 开发者改了地址但没有重新打包 APK

### 问题：手机浏览器打不开 `/health`

常见原因：

- 手机和电脑不在同一个 Wi-Fi
- Windows 防火墙拦了 `8000` 端口
- 填错了 IP 地址
- 后端不是用 `--host 0.0.0.0` 启动的

### 问题：APK 能安装，但 Onboarding 提交失败

常见原因：

- 网络通了，但某个接口调用失败
- 这时应该让开发者去看后端日志

### 问题：远程测试者始终连不上

如果后端只跑在你本机，这是正常现象。

解决方式：

- 部署后端到公网服务器
- 或者用隧道工具暴露本地服务

## 9. 开发者发包前最终检查清单

- 后端能正常启动
- 电脑上 `/health` 正常
- 手机浏览器能打开目标 `/health`
- `API_BASE_URL` 已改成真机可访问地址
- 改地址后已重新打包 APK
- 测试者拿到的是最新 APK
- 测试者知道自己是否必须连同一 Wi-Fi
