# PRD：AI 搭子兼规划 App（Android MVP）

## 1. 产品概述

### 1.1 产品名称（暂定）
- 中文：搭子计划官 / AI 学习搭子 / AI 自律搭子
- 英文：BuddyPlan / AI Accountability Buddy

### 1.2 产品定位
一个面向个人用户的安卓 App：用户设定目标、截止时间、固定安排和希望的“搭子人格”，系统自动生成计划、安排时间、聊天提醒、支持请假改期，并通过打卡证明做基础监督。

### 1.3 一句话描述
“一个像真人搭子一样陪你完成目标的 AI：会安排、会提醒、会商量改期、会催进度、会收打卡。”

## 2. 背景与问题

现实中的学习/锻炼/成长类搭子难找，原因包括：
1. 双方目标不同。
2. 时间表难重合。
3. 长期监督成本高。
4. 关系尴尬：太严格容易不舒服，太松又失去监督效果。

本产品通过 AI 搭子解决“陪伴 + 规划 + 提醒 + 改期 + 打卡监督”的闭环问题。

## 3. 目标用户

### 3.1 核心用户
- 有明确目标但执行不稳定的学生
- 想培养自律习惯的年轻人
- 需要陪伴式监督而不是纯工具型待办的用户
- 有中短期目标（考试、减脂、阅读、课程学习、证书准备）的人

### 3.2 典型场景
1. 备考用户：两个月内准备四六级/考研/法考。
2. 健身用户：六周内养成固定锻炼习惯。
3. 阅读用户：30 天内完成书单。
4. 技能用户：在截止日期前完成课程/项目。

## 4. 产品目标

### 4.1 业务目标（MVP）
- 验证用户是否愿意为“AI 搭子式监督”持续使用
- 验证聊天提醒 + 动态改期 + 打卡证明是否能提高完成率

### 4.2 用户目标
- 更容易开始任务
- 减少拖延
- 在日程变化时快速重排
- 获得陪伴感与成就感

### 4.3 关键指标
- D1/D7 留存
- 目标完成率
- 任务按时完成率
- 打卡提交率
- 请假/改期后最终补完率
- 预警后挽回率（收到风险提醒后仍能按时完成的比例）

## 5. 产品原则

1. **像搭子，不像上级**：语气陪伴式，但规则清晰。
2. **先可落地，再做复杂智能**：MVP 先实现基础规划与监督。
3. **动态重排优先**：现实生活会变化，计划必须可调整。
4. **轻量反作弊**：降低随意打卡，但不追求绝对识别真伪。
5. **支持式监督**：即使判断用户在拖延，也不能羞辱或攻击。

## 6. 功能范围

## 6.1 MVP 功能

### F1. 用户创建 AI 搭子
用户可配置：
- 搭子风格：严肃 / 温柔 / 幽默 / 冷静 / 热血 / 毒舌（轻度）
- 搭子类型：学习 / 健身 / 阅读 / 通用成长
- 提醒强度：轻提醒 / 标准 / 严格
- 表达偏好：简洁 / 聊天感 / 鼓励型

系统根据配置生成一个长期可复用的 Agent Persona。

#### 验收标准
- 用户首次进入可完成搭子创建。
- 系统会生成一段可展示的人格简介。
- 后续聊天和提醒风格与该人格基本一致。

### F2. 用户输入目标与时间边界
用户可输入：
- 目标名称
- 截止时间
- 目标描述
- 每周可投入时长（可选）
- 优先级
- 固定不可用时间（课表/课程/会议/睡觉/通勤）
- 偏好时间段（如晚上效率更高）

#### 验收标准
- 支持创建长期/中期目标。
- 支持输入或编辑固定安排表。
- 支持修改截止时间后重新生成计划。

### F3. AI 自动拆解目标并规划路线
系统根据目标类型自动输出：
- 阶段拆分
- 每阶段任务
- 每个任务的预计时长
- 建议证明方式
- 安排到具体日历空档中

示例：
- 学习目标：复习章节、刷题、错题整理、模拟测试
- 锻炼目标：有氧、力量、拉伸、恢复
- 阅读目标：章节阅读、摘录、复述

#### 验收标准
- 计划结果至少包含未来 7 天任务。
- 每个任务都有名称、时长、开始/结束时间。
- 每个任务都带有建议打卡方式。
- 遇到冲突时自动错开固定日程。

### F4. 聊天式提醒
到达任务开始前/开始时/结束后，系统以聊天消息方式提醒：
- 任务即将开始提醒
- 任务进行中催办
- 截止前提交证明提醒
- 长期风险预警

#### 验收标准
- 至少支持 push + App 内聊天消息双通道。
- 同一任务可配置多个提醒点。
- 提醒文案符合用户设定的人格风格。

### F5. 请假/改期机制
当用户表示“不想做”“想请假”“今天做不了”时，系统：
1. 获取原因（若未提供则追问一句）
2. 判断是合理冲突、疲劳、紧急情况、低动机或明显拖延风险
3. 给出建议：
   - 直接改期
   - 拆小任务
   - 降低难度
   - 改成保底版任务
   - 给出短暂休息再开始
4. 确认后重新规划时间

#### 说明
“确认他不是偷懒”在产品上不应该做成审判式判断，而应实现为：
- 识别**拖延风险**
- 提供**支持式协商**
- 记录改期原因与频率

#### 验收标准
- 用户可在任务详情页和聊天页发起改期。
- 改期后原任务状态更新，新任务自动排入空档。
- 系统会记录改期原因类别与原始文本。

### F6. 任务完成打卡与证明上传
用户完成任务后，需要提交证明。证明方式由 AI 在任务生成时推荐：
- 作业类：上传作业截图
- 阅读类：上传书页/书封照片，或写一句总结
- 运动类：上传运动截图/自拍/短视频
- 难验证类：向 AI 搭子发送一句完成总结，或语音说明

系统对提交材料进行基础分析：
- 是否与任务主题相关
- 是否明显为空白/无关图片
- 是否达到最低提交要求
- 生成简短反馈与建议

#### 验收标准
- 支持图片上传；MVP 可先不做视频/音频。
- 系统能给出“通过 / 存疑 / 需补充”的结果。
- 存疑时要求补一段文字或补一张图。

### F7. 进度预测与风险预警
系统持续根据以下信息判断用户是否赶得上截止时间：
- 已完成任务比例
- 连续改期次数
- 实际完成速度 vs 计划速度
- 剩余可用时间

当风险高时，给出预警：
- 说明为什么赶不上
- 提供补救方案
- 询问是否接受“冲刺模式”计划

#### 验收标准
- 风险评分可从 0-100 输出。
- 风险 > 阈值时自动触发预警消息。
- 预警后用户可一键接受重规划。

## 7. 非目标（当前不做）

- 不做完全开放社交平台
- 不做重度心理治疗类产品
- 不承诺绝对识别用户是否撒谎或作弊
- 不在 MVP 阶段做复杂游戏化系统

## 8. 核心用户流程

### 8.1 首次使用流程
1. 登录/游客进入
2. 选择搭子风格与类型
3. 填写一个目标
4. 输入截止时间与固定安排
5. 生成首版计划
6. 查看今日任务并开始执行

### 8.2 每日执行流程
1. 用户收到聊天提醒
2. 用户开始任务
3. 用户完成后上传证明
4. 系统校验并反馈
5. 更新统计与后续计划

### 8.3 改期流程
1. 用户在提醒消息中点击“今天做不了”
2. 系统询问原因
3. 系统判断改期策略
4. 用户确认新方案
5. 系统重排计划并同步提醒

## 10. 任务与状态定义

### 任务状态
- `planned` 已规划
- `scheduled` 已安排具体时段
- `active` 进行中
- `completed` 已完成
- `missed` 未完成
- `rescheduled` 已改期
- `waived` 本次豁免/取消

### 打卡审核状态
- `pending_review`
- `accepted`
- `needs_more_evidence`
- `rejected`

## 11. 规则设计

### 11.1 规划规则
- 不安排进固定不可用时间
- 尽量尊重用户偏好时间段
- 单个任务超过 90 分钟时优先拆分
- 截止时间越近，任务优先级越高
- 连续失败的任务优先安排成更小颗粒度

### 11.2 改期规则
- 紧急情况：允许直接顺延
- 疲劳/低状态：优先改成保底任务
- 明显拖延：给一个更容易开始的替代任务，不直接放弃整天
- 连续改期过多：触发预警并提示缩减目标或延长截止时间

### 11.3 打卡规则
- 每个任务必须指定至少一种最低证明方式
- 无法有效识别的任务，允许“文字总结 + 自评”兜底
- 重复上传同一素材需标记为可疑，但不要直接指控作弊

## 12. 风险与约束

### 12.1 产品风险
- 用户可能反感“被 AI 审核”
- 证明材料识别准确率有限
- 提醒太频繁会造成骚扰
- 人格风格过强可能引发不适

### 12.2 应对策略
- 设置提醒强度与人格强度
- 提供人工确认式补充说明
- 提供隐私设置与数据删除能力
- 所有风险判断都提供解释与申诉入口

## 13. 非功能性要求

### 性能
- 常规接口 < 500ms
- 计划生成 < 8s
- 打卡审核 < 10s

### 稳定性
- 推送失败时 App 内仍可见未读提醒
- 服务端任务失败可重试

### 安全与隐私
- 图片、日程、聊天数据默认加密存储
- 支持用户删除打卡素材
- 支持配置素材保留期限

## 14. MVP 成功标准

满足以下条件可认为 MVP 可验证：
1. 用户能在 5 分钟内完成从注册到生成首版计划。
2. 用户能在一周内持续收到提醒并完成至少 3 次有效打卡。
3. 系统能处理至少 1 次改期并成功重排。
4. 系统能在明显进度落后时触发预警。
5. 用户主观上感受到“像搭子在陪我”，而不是普通待办工具。
